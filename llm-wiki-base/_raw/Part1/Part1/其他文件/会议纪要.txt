## 报警求助模式与场景路由规划设计方案

### 一、设计背景与目标

在真实报警接处警场景中，报警人往往无法在第一时间完整、准确地描述警情类型（如究竟是普通民宅火灾，还是涉及危险品的工业生产火灾）。传统的固定问答流程容易导致：

- **效率低下**：无论何种警情都走同一套长流程，简单警情耗时过长。
- **场景固化**：一旦初始判断错误，后续难以修正关键信息。
- **知识缺失**：特定场景（如危化品泄漏、水域救援）需要即时提供专业指导，却未能及时触发。

本方案提出 **“极简入门 + 动态场景路由”** 的智能对话策略，实现：

1. **快速起步**：基于高概率场景加载极简模板，秒级获取核心要素。
2. **动态修正**：实时识别报警人语音中的热词，自动跳转到更匹配的场景集或具体子场景。
3. **知识增强**：跳转后立即关联对应场景的风险提示、处置规范、设备清单等知识库。

### 三、6大基础场景集及极简模板

| 场景集 ID | 名称 | 典型关键词 | 极简模板（必问项） |
|-----------|------|------------|-------------------|
| S01 | 普通火灾 | 着火、冒烟、火光 | 起火物？有无被困？有无爆炸危险？ |
| S02 | 危化品事故 | 泄漏、化学、燃气、汽油 | 什么物质？有无毒害/爆炸？是否需要防化？ |
| S03 | 交通事故救援 | 车祸、追尾、侧翻 | 车辆类型？有无人员被困？有无起火？ |
| S04 | 水域救援 | 溺水、落水、洪水 | 落水人数？水域类型（河流/湖泊/海）？ |
| S05 | 坍塌/建筑物事故 | 倒塌、塌方、埋压 | 被困人数？结构类型（建筑/土方）？ |
| S06 | 人员被困/特殊救助 | 电梯困人、反锁、跳楼 | 被困位置？有无生命危险？ |

> **缺省模式**：若无法匹配以上任何场景集，默认加载“通用救助”极简模板，同时加强热词监听。

### 四、热词识别与动态路由规则

系统在对话全程运行**热词扫描器**，一旦命中预设的热词，立即触发路由跳转。

#### 4.1 热词 → 目标场景映射表（示例）

| 热词（正则/包含） | 命中后目标 | 跳转类型 | 附加动作 |
|------------------|------------|----------|----------|
| 医院、学校、商场、客运站、地铁 | 人员密集场所场景集 | 场景集跳转 | 加载疏散指导知识库 |
| 汽油、柴油、液化气、天然气、化学品 | 危化品场景集 | 场景集跳转 | 加载安全距离+防爆知识 |
| 隧道、高架、高速、立交 | 交通事故特殊场景 | 直接进入子场景 | 加载隧道火灾/连环追尾模板 |
| 地下室、地库、地下车库 | 地下空间火灾 | 场景内细化 | 加载排烟、搜救特殊设备 |
| 老人、小孩、残疾人 | 弱势群体被困 | 子场景细化 | 追加医疗急救知识块 |
| 爆炸、轰然、巨响 | 爆炸风险 | 预警升级 | 立即推送防爆指令 & 调派增援 |

#### 4.2 跳转决策逻辑（伪代码）

```python
def on_hotword_detected(hotword, current_scene_set, current_sub_scene):
    target = mapping_table.get(hotword)
    if target.type == "scene_set":
        # 跳转到完全不同的场景集
        switch_to_scene_set(target.id)
        reset_questionnaire(target.default_template)
    elif target.type == "sub_scene":
        # 在当前场景集内细化
        if current_scene_set == target.parent_set:
            drill_down_to_sub_scene(target.id)
        else:
            # 跨场景集细化：先切换场景集，再加载子场景
            switch_to_scene_set(target.parent_set)
            drill_down_to_sub_scene(target.id)
    # 附加知识库推送
    if target.knowledge_base:
        push_knowledge(target.knowledge_base)
```

### 五、典型路由示例

#### 示例1：普通火灾 → 危化

- **初始**：报警人说“厨房着火了”，系统匹配普通火灾（S01），加载极简模板，询问“起火物是什么？”  
- **报警人回答**：“我家里存了几桶汽油……”  
- **热词触发**：汽油 → 映射到危化场景集（S02）  
- **系统动作**：立即跳转，加载危化问答模板，并主动提示“请所有人撤离到上风向，不要接近明火”。  
- **后续**：继续追问汽油存量、有无其他化学品等。

#### 示例2：缺省模式 → 人员密集场所

- **初始**：报警人说“这里冒烟了”，无法明确场景，进入缺省模式。  
- **对话中**：“我们在XX医院三楼……”  
- **热词触发**：医院 → 跳转到人员密集场所场景集。  
- **系统动作**：加载医院火灾专项模板（询问楼层、病人数量、疏散通道等），并推送院方应急广播知识。

#### 示例3：交通事故 → 危化

- **初始**：“高速上两车追尾”，进入交通事故救援（S03）。  
- **进一步描述**：“有一辆是油罐车，在漏油……”  
- **热词触发**：油罐车、漏油 → 跳转至危化场景集，且直接加载“油罐车泄漏”子场景。  
- **系统动作**：追加危化事故调派标准（防化车、泡沫车、警戒范围）。

### 六、场景级与场景的细分设计

为避免过度跳转导致混乱，系统定义两级结构：

- **场景级（Scene Set）**：上述6大类别，每个场景级包含：
  - 默认极简模板
  - 一组可跳转进入的子场景
  - 关联的知识库索引

- **子场景（Sub Scene）**：具体可操作的模板，例如：
  - 危化场景级下的子场景：燃气泄漏、汽油火灾、氯气泄漏、锂电池火灾……
  - 交通事故下的子场景：小车追尾、大巴侧翻、危化车泄漏、隧道事故……

**规则**：
1. 如果直接命中子场景热词（如“锂电池”），且系统已处于对应的场景级（如危化），则直接加载该子场景模板，不再过问场景级通用问题。
2. 如果命中子场景热词但当前处在不相关的场景级（如在水域救援中听到“锂电池”），则先切换场景级再到子场景。
3. 如果仅命中场景级热词（如“化学”），则切换到危化场景级，加载其极简模板，之后再通过后续热词或逻辑判断进一步细化。

### 七、知识库联动设计

当路由跳转到特定场景级或子场景时，系统应主动推送以下知识（以语音或文本卡片形式）：

| 场景 | 推送内容示例 |
|------|--------------|
| 危化品泄漏 | 安全距离（至少300米）、风向判断、消防泡沫准备、防化服穿戴提醒 |
| 高层建筑火灾 | 疏散楼梯位置、禁止使用电梯、云梯车停靠点建议 |
| 人员密集场所 | 应急广播词、疏散引导方法、防踩踏措施 |
| 水域救援 | 抛绳包使用、冲锋舟组装时长、溺水者保暖 |
| 交通事故（高速） | 后方警示距离（150米以上）、防闯入预警、路障设置 |

知识库内容可预先结构化存储，并通过TTS实时播报或发送至接警员终端。

### 八、实施建议与后续优化

#### 8.1 热词库建设
- **基础热词**：从历史警情数据中提取高频词，人工标注目标场景。
- **动态扩展**：支持通过管理后台添加新热词，并配置跳转规则。
- **歧义处理**：对于一词多义（如“炸弹”可能是恐袭或危化爆炸），设计优先级或二次确认逻辑。

#### 8.2 性能要求
- 热词识别延迟 < 500ms，避免跳转滞后导致无效提问。
- 场景切换时需保留已收集的有效信息（如地址、联系方式），避免重复询问。

#### 8.3 测试用例
建议准备至少20个混合场景进行模拟测试，例如：
- 初始描述为“冒烟”，中途提及“医院”“化学品”“爆炸”等，验证路由是否正确切换。

#### 8.4 人工兜底
当热词触发错误或跳转出现循环时，提供“强制重置”按钮，允许接警员手动选择场景级并锁定。

# 大模型辅助规则持续迭代方案（LLM-augmented Rule Evolution）

**来源**：119消防接警机器人系统（火灾+开锁/马蜂窝等社会救助场景）
**版本**：V1.0

---

## 1. 可行性分析

### 技术成熟度
- LLM在规则提取、规则提炼、规则自迭代领域已有大量成熟案例（Rule Induction + LLM-in-the-Loop）
- 大模型可从真实呼救日志中自动发现新同义词、反义信号、新紧急条件或规范漏洞
- 生成的SWRL规则或SHACL约束可追加到现有本体，不破坏原有结构

### 场景适配度
- 社会救助类型多样且口语化强（"进不去屋""蜂飞得我不敢开门"）
- 消防规范动态演化（"减副"、分流优先、夜间马蜂窝处置）
- 已有的本体（RDF/OWL + SHACL/SWRL）天然支持迭代

### 实际收益量化
- 规则覆盖率：初始60-70% → 3-6月后92-97%
- 槽位F1提升5-12%
- 误派车率下降15-25%
- 6人团队工作量降低40-60%

## 2. 优缺点对比

| 维度 | 本方案(大模型持续迭代规则) | 纯手工规则 | 一次性大模型蒸馏 |
|------|---------------------------|-----------|-----------------|
| 可解释性 | 高（规则+CoT日志） | 最高 | 中（灰盒） |
| 规则更新速度 | 快（每周/每月自动迭代） | 慢（人工手动） | 无迭代 |
| 适应新口语/新场景 | 极强 | 弱 | 中 |
| 幻觉风险 | 低（SHACL验证+人工审核） | 无 | 中 |
| 人力投入 | 中（6人团队只做审核） | 高 | 低（一次性） |
| 维护成本 | 低（自动化闭环） | 高 | 中 |
| 消防审计友好度 | 最高 | 高 | 中 |

**最大优势**：实现"人机协同闭环"——大模型发现潜在规则，6人团队把关，系统越用越聪明

## 3. 具体迭代流程（8周一个完整闭环，可每周小迭代）

### Step 1: 数据收集（自动）
机器人上线后，实时记录所有呼救日志（ASR转录 + 当前问题 + 小模型输出 + 最终人工/调度结果）

### Step 2: Teacher大模型分析（每周/每月批量运行）

**Prompt模板**：
```
你是119规则提炼专家。当前日志批次：
{log_batch}

请严格遵守现有本体与消防规范（分流优先、夜间马蜂窝、安全评估）。

步骤：
1. 找出新同义词/反义信号/新紧急条件（开锁/马蜂窝示例）
2. 提出新SWRL规则或SHACL约束建议
3. 输出可直接导入的Turtle格式规则 + CoT解释 + 置信度
4. 标记需人工审核的边缘案例
```

### Step 3: 6人团队审核（核心人力投入）
- 只审核大模型提出的新规则（而非全量标注）
- 重点校验：是否符合消防规范？同义词是否准确？反义判断是否合理？
- 通过率>85%后自动导入本体

### Step 4: 规则更新与重新蒸馏（自动）
- 新规则追加到GraphDB
- 触发小模型增量蒸馏（只更新规则相关部分，LoRA高效）
- 可选：多Agent中Filter-Orchestrator优先使用最新规则

### Step 5: 上线验证（A/B测试）
- 新规则灰度上线，监控槽位F1、误派车率、分流率
- 闭环结束，进入下一轮

## 4. 迭代示例

### 开锁示例迭代
- **第1轮**：发现新同义"门打不开""钥匙忘在屋里"
- **第2轮**：发现反义信号"有备用钥匙" → 自动降低优先级+建议联系物业
- **新规则**：`OpenLock(?e) ^ hasSpareKey(?e, true) → urgencyLevel(?e, "low") ^ suggestedAction(?e, "分流物业")`

### 马蜂窝示例迭代
- **第1轮**：发现"蜂飞得厉害"同义
- **第2轮**：发现"夜间处理"偏好 → 新SWRL：`WaspNest(?e) ^ hasHighLocation(?e, true) → suggestedTime(?e, "夜间优先")`

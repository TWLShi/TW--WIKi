# 消防119接警系统中火灾救援与社会救助的复用与差异分析

**来源**：针对中国大陆119消防接警（含机器人接警）场景
**版本**：V1.0

---

## 1. 可以高度复用的部分（80%以上直接共享）

### 接警基础流程与机器人交互框架
- 单问题引导式对话（"位置在哪？"、"有没有人需要帮助？"、"现场情况怎么样？"）
- ASR转录 + 噪声处理 + 无关内容过滤
- 机器人语音播报、情绪安抚、多轮追问机制
- 端到端延迟控制、合规审计（PROV-O溯源）

### 核心本体与知识框架
- 基础类：`DisasterEvent`、`hasLocation`（位置 + 区域特征：地下室、高层、电梯内等）
- 通用关系：上位/下位、整体/部分（hasPart）、因果（causes）、蕴含（implies）
- SHACL约束（必须有时空信息）、SWRL基础规则（传递性因果）
- 标注指南核心结构、6人团队标注流程、Label Studio配置
- RDF输出格式、调度系统API接口（推送JSON画像到GIS/CTI）

### 模型与蒸馏基础
- 单小模型主路径（位置槽位最通用）
- 蒸馏数据集准备、Few-shot Prompt模板（先验知识注入）
- 轻量Graph RAG知识库构建（本体 + 历史案例）
- 边缘部署（ONNX量化）、云边混合架构

### 通用槽位与实体维度
- 位置槽位（地址 + 区域特征：低洼/高层/电梯内）
- 受困/求助人数 + 基本状态（有人/无法出去）
- 相关性过滤、无关内容标记
- 置信度输出 + 低置信转人工机制

**复用收益**：开发工作量减少60-70%。机器人硬件、ASR、调度对接几乎100%复用。

## 2. 需要不同/新增的工作内容（差异化重点，约30-40%增量）

### 本体与知识框架扩展（必须新增）

**新增子类**：
- `SocialRescueEvent`（社会救助事件）
- `ElevatorTrapped`（电梯困人）
- `GasLeak`（煤气泄漏）
- `ElderFall`（老人摔倒）
- `AnimalRescue`（动物救助）

**新增属性**：
- 求助类型（电梯/煤气/摔倒）
- 紧急程度分级（非火灾高优先级判断）
- 所需资源类型（特勤车/破拆工具/医疗 vs 消防车/水枪）

**新增SWRL规则**：
- "电梯困人 + 高层" → 优先级高
- "煤气泄漏" → causes 爆炸风险（与火灾烟火因果不同）

**SHACL调整**：社会救助需更多"求助状态"约束，而非"烟火程度"

### 槽位与实体标注维度扩展

**新增/细化槽位**：
- 紧急类型（火灾 vs 社会救助子类型）
- 干预需求（破拆/医疗/开门）
- 细微事实：
  - "电梯门缝卡住" vs "突然停电"
  - "老人胸痛伴摔倒"

**标注指南补充**：社会救助子类型列表、专项span标注（每个Agent对应一列）

**数据准备**：社会救助录音占比至少50%，生成专项合成数据

### 多Agent设计与蒸馏差异

| Agent | 复用/差异 | 说明 |
|--------|-----------|------|
| Position-Agent | 复用 | 位置几乎相同 |
| Help-Agent | 需扩展 | 社会救助专用：人数 + 求助状态 + 人群类型：老人/儿童/动物 |
| Type-Agent | 需强化 | 区分"火灾烟火" vs "社会救助类型 + 程度" |
| Filter-Orchestrator | 需新增逻辑 | 火灾 vs 社会救助分类 + 资源建议（误派车风险控制） |

**蒸馏工作量**：每个Agent准备专项数据集，社会救助案例更多样，标注工作量增加30-40%

### 机器人交互与决策逻辑差异

| 场景 | 重点问题 | 机器人模板 |
|------|----------|------------|
| 火灾 | 烟火程度、燃烧物、被困位置 | "灭火准备" |
| 社会救助 | 求助具体状态、安抚情绪、确认非火灾 | "工具/人员/医疗准备" |

**优先级判断**：社会救助常需快速判断是否转120医疗或物业联动

### 资源调派与输出画像差异

| 维度 | 火灾 | 社会救助 |
|------|------|----------|
| 调派资源 | 消防车、水枪、破拆 | 特勤车、电梯救援工具、医疗或社会力量 |
| RDF新增字段 | - | `rescueType`（fire vs social）、`requiredResource` |
| 联动部门 | 内部为主 | 物业、特种设备、民政 |

### 测试与上线差异
- 单独准备社会救助测试集（高频类型：电梯困人、煤气泄漏、老人救助）
- A/B测试重点验证"火灾 vs 社会救助"分类准确率和误派车率
- 社会救助更注重交互自然度（情绪安抚）和多轮确认

## 3. 开发工作量建议（基于6人团队）

| 工作类型 | 比例 | 说明 |
|----------|------|------|
| 复用工作 | 60-70% | 本体基础、单小模型、ASR、机器人硬件对接、通用标注工具、Few-shot Prompt框架 |
| 新增/差异工作 | 30-40% | 本体扩展、专项标注、多Agent扩展/新Agent蒸馏、机器人交互模板、专项测试 |

**新增工作细分**：
- 本体扩展 + 新SWRL/SHACL（1-2周）
- 社会救助专项标注数据（6人团队2-3周，覆盖高频类型）
- 多Agent扩展/新Agent蒸馏（2周）
- 机器人交互模板 + 决策逻辑调整（1-2周）
- 专项测试集与上线灰度

**总体建议**：先完成火灾核心系统，然后通过**增量迭代**方式添加社会救助模块（复用80%代码，只新增本体分支 + 专项数据）。

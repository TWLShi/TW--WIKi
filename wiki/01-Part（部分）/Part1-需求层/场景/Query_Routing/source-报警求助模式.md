# 报警求助模式与Query_Routing

*摘要：本文档深入分析消防调派智能系统中报警人在紧急情况下的求助行为模式，以及 Query_Routing（智能问询路由）如何通过 DIKW 框架将这些非结构化描述转化为结构化决策知识。*

## 核心概念

### 1. 常见描述模式

报警人（市民）在极度紧张或紧急情况下描述灾情时呈现出的语言习惯和信息组织方式^[inferred]：

- **信息提供逻辑**：分析在不同火灾（如高层、化工）或救援场景（如洪水）下，报警人通常会主动、优先提供哪些信息
- **语言习惯识别**：识别报警人在极度压力下可能使用的非规范化描述（如对火势大小、烟雾颜色的形容模式）

### 2. 关键缺失要素

"标准全要素模板"与"常见描述模式"之间的差值^[inferred]：

```
关键缺失要素 = 该场景的标准全要素模板 - 报警人的常见/实际描述模式
```

**为什么重要**：识别那些对调派规模计算和风险研判至关重要，但报警人并未在第一时间主动提及的信息碎片

### 3. 关键描述模式（标准模板）

由 12类子场景画像定义的决策闭环所需的所有核心参数^[extracted]：
- 例如：高层建筑的起火楼层、被困人员具体位置、是否有易燃易爆品等

## 逻辑闭环

四个概念共同构成接警阶段的智能问询决策链：

```
输入端（常见描述模式）
    ↓
对比标尺（关键描述模式/标准模板）
    ↓
逻辑计算（关键缺失要素）
    ↓
执行端（问询问题集的路由）
```

## Query_Routing 在 12 步流程中的核心作用

在核心调派引擎的 **Step 3（智能问询）** 阶段，路由机制扮演着"交通枢纽"的角色^[inferred]：

1. **标签触发**：系统利用 DIKW 框架 将报警人的实时描述转化为标签（Tags）
2. **差额检索**：通过对比 MOC-子场景画像 中的决策必填项，系统自动勾勒出哪些关键字段为空
3. **动态路由**：Query_Routing 不仅仅是列出问题，它会根据当前已掌握的信息（信息熵），动态计算最能缩减决策不确定性的问询路径

## DIKW 知识转化

| 层级 | 内容 |
|------|------|
| 数据 | 原始报警音频或文本 |
| 信息 | 识别为特定的描述模式 |
| 知识 | 沉淀为"子场景画像库" |
| 智慧 | AI 多智能体决策 |

## 阅读建议

- 业务人员：重点阅读"子场景画像"，理解不同灾害类型下报警人的心理与描述特征
- 开发人员：需参考这些模式来设计数据模型，确保系统能捕捉并处理这些非结构化的求助信息
